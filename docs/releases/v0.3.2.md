## User-Facing Changes

### Rigbundles First-Class

`rig bundle` now ships first-class cross-primitive bundling:

```
rig bundle create <spec> -o <path>
    [--name <n>] [--bundle-version <v>] [--include-packages <refs...>]
    [--rig-root <r>] [--notes <text>]
    [--min-daemon-version <ver>] [--min-cli-version <ver>] [--json]
rig bundle inspect <path> [--json]
rig bundle install <path>
    [--plan] [--yes] [--target <root>]
    [--skip-version-check] [--force] [--json]
rig bundle history [--rig <name>] [--since <iso>] [--json]
```

Content kinds routed at install: `skills`, `plugins` (hybrid), `workflow_specs`,
`context_packs`, `agent_images`. Each kind lands in its canonical
library under `$OPENRIG_HOME`; consumer-scan visibility is preserved.

`bundle.yaml` author manifests are auto-detected when present.
Vendoring uses both-sides path containment (source and destination)
and symlink-escape protection. Integrity hashing is captured in the
bundle manifest.

`--skip-version-check` and `--force` on `install` are operator-explicit
overrides for the install-time daemon/CLI compatibility gate and the
install-time conflict gate respectively (NOT recommended for routine
use; the overrides exist for known-good operator scenarios).

`rig bundle history` reads `~/.openrig/bundle-audit.jsonl` â€” the
audit ledger of every install â€” with optional filters on
target rig name and earliest install time.

### Workspace Doctor (`rig workspace doctor`)

7-check workspace-readiness diagnostic against the daemon's resolved
workspace:

```
rig workspace doctor [--workspace <path>] [--strict] [--json]
```

Checks: workspace root, missions folder, file allowlist, daemon
alignment, daemon reload, optional slice docs, MISSION_NOTES presence.

Default exit-code: non-zero only on `fail`. `--strict` makes
warn-or-fail non-zero. The CLI overlays `OPENRIG_FILES_ALLOWLIST`
from the operator's shell env so the diagnostic reflects the
operator's intended allowlist even when the daemon's env is unchanged.

### Queue Ergonomics

`rig queue create` accepts a body from a file or stdin and exposes
first-class mission/slice scope:

```
rig queue create --source <s> --destination <d> \
    (--body <text> | --body-file <path>) \
    [--mission <id>] [--slice <id>] \
    [--priority <p>] [--tier <t>] [--tags <csv>] \
    [--target-repo <name>] [--no-nudge] \
    [--expires-at <iso>] [--id <qitemId>] [--json]
```

`--body-file -` reads from stdin. `--mission <id>` / `--slice <id>`
translate to `mission:<id>` / `slice:<id>` tags (compose with
`--tags`).

### Operator Context-Mode Bindings (`rig policy`)

```
rig policy set <mode> [--scope <scope>] [--qualifier <id>] \
    [--<field> ...] [--evidence <citation>] \
    [--confirm] [--bearer <token>] [--json]
rig policy show [--json]
rig policy effective [--rig <id>] [--workstream <id>] [--qitem <id>] [--json]
rig policy cite [--rig <id>] [--workstream <id>] [--qitem <id>]
rig policy unset <scope> [qualifier] [--bearer <token>] [--json]
rig policy defaults [--json]
```

`set` is restate-only without `--confirm` (the CLI echoes the proposed
binding and exits with `exit 2` so scripts cannot silently apply).

Six modes: `sleep | desk | mobile | away | focus | debug`. Four
scopes: `global_host | rig | workstream | qitem`. `--qualifier` is
required for `rig | workstream | qitem` scopes and explicitly
rejected for `global_host`.

Operator-edit verbs (`set --confirm`, `unset`) require an operator
bearer token (`--bearer` or `OPENRIG_AUTH_BEARER_TOKEN` env).

### Scope Tree Primitive (`rig scope`)

```
rig scope slice ls    [--mission <m>] [--state <s>] [--json]
rig scope slice show  <slice-path> [--mission <m>] [--json]
rig scope slice create <mission> <slug> [--template <k>] [--title <t>] [--json]
rig scope slice ship  <slice-path> <release-mission> [--mission <m>] [--json]
rig scope slice close <slice-path> --reason <r> [--note <t>] [--mission <m>] [--json]
rig scope slice move  <slice-path> <dest-mission> [--mission <m>] [--json]
rig scope mission ls  [--json]
rig scope mission show <mission> [--json]
rig scope mission create <name> [--template <k>] [--id <dot-id>] [--title <t>] [--json]
```

Top-level option `--workspace <path>` overrides workspace root
(otherwise inferred from `cwd` or `$OPENRIG_WORK_ROOT`).

### `rig up` Paper-Cut Fixes

```
rig up <source> [--plan] [--yes] [--cwd <path>] [--target <root>] [--existing] [--json]
```

- Pre-launch failures now return structured HTTP 4xx
  (`cycle_error` / `preflight_failed` / `validation_failed` /
  `service_boot_failed`) instead of bare 500.
- Failed boots no longer leave orphan rig records on disk.
- Path-form invocations default `cwd` to the caller directory.
- `--existing` skips library-spec name resolution and treats
  `<source>` as an existing rig name directly.

### MISSION_NOTES Durable Pattern (slice 21 FR-3)

`rig scope mission create` auto-scaffolds `MISSION_NOTES.md` from
`conventions/mission-notes/TEMPLATE.md` by default; `--no-mission-notes`
opts out (rare). The convention codifies the durable-pattern model
documented in workspace skills + canonical templates so missions
ship with a uniform top-of-mind notes surface.

## Operator / Setup Notes

After upgrading the published CLI:

```bash
npm install -g @openrig/cli@0.3.2
rig --version    # expect 0.3.2
rig daemon stop
rig daemon start
```

Migration `041_rig_policy` applies automatically on first post-upgrade
daemon start. It is a non-destructive `CREATE TABLE` with no impact on
existing data; binding rows are operator-authored at runtime through
`rig policy set --confirm`.

For the new `rig policy` surface: opt-in default-none. With no
bindings set, no behavior change. Use `rig policy defaults` to inspect
the recommended per-mode 6Ã—7 + default-scope mapping before authoring.

For the new `rig scope` surface: read-only against workspace by
default (`ls`/`show`); authoring verbs (`create`/`ship`/`close`/`move`)
operate on the substrate scope tree as advertised. Use `--workspace
<path>` to override the resolved workspace root.
