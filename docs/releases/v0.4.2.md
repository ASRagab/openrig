# OpenRig v0.4.2

## Summary

OpenRig v0.4.2 is a targeted hotfix for the CLI `daemon-status` probe's
false-negative window right after a `rig daemon start` / restart. On the
prior 0.4.1 build, the probe issued a single `/healthz` fetch per
`getDaemonStatus` branch; if the daemon process was up but the HTTP
listener had not yet begun accepting, the single probe lost the race and
`rig daemon status` reported a false "Daemon not running" /
"healthz failed" even though `/healthz` was returning `200`.

The fix wraps the three `getDaemonStatus` `/healthz` branches in a
status-probe-local bounded settle: max 5 attempts, 200ms backoff,
hard-bounded by construction. A genuine-down still fails all attempts
and is reported stopped / unhealthy after the budget — never masked,
never an unbounded wait. `fetchDaemonProbe` and start / stop / checkPid
timing are unchanged.

## What Changed

- **`rig daemon status` no longer false-negatives during the daemon's
  post-start listener-bind window.** The probe re-checks up to 5 times
  with a short backoff before it reports stopped, so a healthy daemon
  that is still binding its listener resolves as running / healthy
  instead of reporting a false down state.
- **Genuine-down reporting is unchanged.** If the daemon really is not
  running, all attempts fail and the probe reports stopped or unhealthy
  after the bounded budget — no hang, no mask.

## Files Touched

- `packages/cli/src/daemon-lifecycle.ts` — three probe branches
  (pid-alive `/healthz`, no-state `/healthz`, and the injected-probe
  wrapper) now go through `probeHealthzWithSettle`.
- `packages/cli/test/daemon-lifecycle.test.ts` — focused probe-layer
  tests: transient-then-healthy settles to running / healthy;
  pid-alive-but-never-answers stays `healthy: false`; genuine-down is
  reported stopped after the hard-bounded budget.

## Behavior + Compatibility

- **No CLI surface changes.** All commands, flags, and arguments are
  identical to `0.4.1`.
- **No daemon / DB / migration changes.** Migration state is unchanged
  from `0.4.1`; no new schema migrations ship in this release.
- **No dependency changes.** `package-lock.json` is refreshed for the
  version bumps only.

Existing v0.4.1 databases are unaffected. Upgrading is simply
`npm install -g @openrig/cli@0.4.2` (or a package refresh) and a normal
`rig daemon` restart if the daemon is running.
