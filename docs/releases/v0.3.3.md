# OpenRig v0.3.3

## Summary

OpenRig v0.3.3 is a usability + honesty release. Two new top-level CLI
verbs land (`rig archive`/`rig unarchive` and `rig add`/`rig add-member`),
several long-standing paper cuts are retired (large-payload startup
delivery, dash-prefixed `send_text`, cmux 0.64.x compatibility, `rig down`
by name, golden-path discovery), and three honesty surfaces are tightened
to stop misclassifying ambiguous outcomes as success (`rig send --verify`
delivery outcomes, Codex restore gates, `rig whoami` peers contract). The
For-You feed gets a manage-by-exception pass (drill-to-terminal,
decision-band sort, one-click approve). Slice 13's deterministic
release-surface parser + skill-binding index ship as product code in
service of the auto-feature-rollout system.

One new migration in 0.3.3: `042_rig_archive`
(`ALTER TABLE rigs ADD COLUMN archived_at TEXT` + `idx_rigs_archived`).
Append-only, non-destructive, no rigs-row restructure. Existing v0.3.2
databases upgrade on first `rig daemon start`.

## Included In This Release

- **Large Startup-File Transport (slice 16)** — `TmuxAdapter.sendText`
  payloads over 100KB now travel via a per-call tmux buffer
  (`load-buffer` + `paste-buffer -d -r`) instead of being embedded in
  a single `send-keys -l` argv. Closes the >128KB silent-launch-fail
  class on Linux (`MAX_ARG_STRLEN`) for large per-seat startup packs.
  `-r` preserves raw LF so multi-line packs do not early-submit on
  every newline; the single trailing C-m stays the caller's separate
  submit. Small payloads keep the exact inline `send-keys -l` command.
  Temp file + buffer cleanup on success AND paste-error paths; unique
  names per call so parallel `rig up` seats do not collide.
- **`send_text` Inline Dash Sentinel (slice 17)** — inline
  `tmux send-keys -t <t> -l <text>` now carries a `--` end-of-options
  sentinel so any startup payload whose content begins with `-`
  (notably `---` YAML frontmatter — the norm for per-seat packs) is
  not parsed by tmux as flags. The slice-16 large/buffer path is
  already immune (content travels via a file, never argv) and is
  untouched. Inert for non-dash content.
- **cmux 0.64.x Compatibility (slice 18)** — cmux 0.64.x renamed the
  surface identifier (`list-panels --json` rows carry `ref`, not
  `id`). The transport's `surface.list` normalizer remapped the array
  key but left row keys untouched, so `surface.sendText` silently
  failed under the new cmux line. Fixed via a `normalizeSurfaceRow`
  helper that resolves the handle from `ref ?? surface_ref ??
  surface_id ?? id`, normalizing every row regardless of array key.
  One resolution order serves both cmux 0.64.x and 0.63.x — no
  version-negotiation shim. Adaptation via the same pattern the code
  already uses for workspace rows and create/split handles.
- **`rig archive` Affordance (slice 19)** — new top-level commands
  `rig archive <rigId>` + `rig unarchive <rigId>`. Archive is
  non-destructive: rig row is preserved, `archived_at` is set, and a
  `rig.archived` event fires. Default reads exclude archived (no
  surprise listings); `rig ps --include-archived`, `?includeArchived=
  true`, and `?archived=only` opt them back in. The UI gains a lazy
  collapsible "Archive" section under the localhost host node, fed
  by a separate archived-only query. `rig up <name>` against an
  archived-only name is refused honestly with an error pointing at
  `rig unarchive` — no silent restore, no silent fall-through. The
  running/degraded archive guard surfaces a 3-part fact / consequence
  / action error and accepts `--force` to override. UI query
  invalidation consumes `rig.archived` / `rig.unarchived` events
  end-to-end so a CLI archive in one window reactively refreshes
  another mounted UI.
- **`rig add` / `rig add-member` (slice 24)** — new top-level command
  for the `add_member` converge op. Adds a member to an existing pod
  in a live rig from a YAML/JSON member fragment file. Built on the
  new converge spine (`Op` union + differ + `convergeOp`), with
  imperative CLI sugar over the converge interface (not a bypass).
  Daemon route `POST /api/rigs/:rigId/pods/:podNamespace/members`
  maps `AddMemberOutcome` codes to HTTP (404 / 409 / 400 / 201).
  MCP tool `rig_add` ships in lockstep. Edge validation tightened in
  the same slice: non-array `edges` is rejected (not silently
  dropped); edge kinds are validated against the canonical
  `VALID_EDGE_KINDS` set exported from `rigspec-schema.ts` and
  reused (not duplicated), so `rig add` edges share the existing
  graph contract.
- **`rig down` Accepts Name-or-Id (slice 22)** — the v0.2.0-era
  `rig down <name>` HTTP 404 caveat is gone. The resolver now fetches
  `/api/rigs/summary?includeArchived=true` and id-matches across ALL
  rigs (archived ids still reach teardown), then name-matches ACTIVE
  rigs only. Active+archived same-name pair resolves the active rig;
  archived-only name does not resolve by name. The packaged
  `openrig-user` skill (specs source + assets/plugins copy +
  `_canonical` mirror) and the `down.ts` resolver comment are
  corrected to reflect the real behavior. Historical CHANGELOG /
  v0.3.1 / v0.3.2 release notes are left intact — the bug genuinely
  existed at those releases.
- **Honest Codex Restore Gate (slice 21 FR-2)** — `verifyResumeLaunch`
  no longer returns `ok:true` unless the native-resume-probe proves
  the seat actually resumed. Unresolved operator-action gates (an
  update that cannot auto-dismiss, trust, model-selection) and a
  bounded poll that never reaches `resumed` now return `ok:false`
  with `recovery: attention_required` and last-12-line pane evidence,
  routed into the existing
  `HarnessLaunchResult` -> startup-orchestrator
  `startupStatus: attention_required` -> `ps`/`status` projection —
  no new state machinery. The working auto-dismiss path is preserved
  (a skippable update gate still auto-dismisses and continues to
  success); only UNRESOLVED gates fail loudly. Resolves the 04.3
  clean-VM `kernel_state bootstrap_failed` failure-mode the herder
  surfaced.
- **`rig send --verify` Honest Delivery Outcomes (slice 99.0.6.3)** —
  the prior `Verified: no` collapsed two materially different states
  into one: a message that LANDED (text pasted + Enter submitted,
  both succeeded) but whose post-send capture raced a TUI redraw,
  and a message that genuinely did not land. Careful agents paid a
  manual `rig capture` roundtrip on every send; careless agents read
  no as failure and double-sent. The three outcomes are now named in
  the response:
  - `delivered` — verified succeeded.
  - `rendered-unconfirmed` — text + Enter both succeeded but the
    capture could not re-confirm (redraw race, or capture threw).
    Landed-but-unconfirmable, NOT failure.
  - `failed` — the transport itself failed.
  The legacy `Verified: yes/no` line is preserved verbatim
  (parsers); a new `Delivery:` line carries the named outcome plus,
  for the middle case, a `rig capture <session>` confirmation
  pointer. `--json` carries the field through automatically.
- **`rig whoami` Peers Contract (slice 99.0.6.1)** — the `peers[]`
  field was readable as a directionally-edged subset or as host
  inventory; it is neither. The grounded contract is now stated
  in-band: `peers[]` is this rig's roster EXCLUDING self (built from
  `rig.nodes` with only a self-skip — no edge/status/binding
  filter), `edges{}` is the directional graph, and
  `rig ps --nodes` is the inventory including self + live state.
  Additive clarification only — no behavior or shape change.
  `WhoamiResult` gains a required `peersNote` field carrying the
  three pointers; the CLI Peers header keeps the literal `Peers:`
  prefix verbatim (shipped tests grep on it) and adds the in-band
  clarifier.
- **Workflow Instantiate By Name (slice 04.1)** — `rig workflow
  instantiate <built-in-name>` (e.g. `conveyor`, `basic-loop`)
  used to fail with `spec_file_missing` because instantiate fed the
  bare name straight to `WorkflowSpecCache.readThrough` (which
  treats its argument as a literal file path). Only a raw
  sourcePath worked — a hidden path that violated the
  fresh-operator goal: discovery advertised names you then could
  not use. `WorkflowRuntime.instantiate` now resolves the passed
  identifier against the seeded spec cache BY NAME first, falling
  back to literal-sourcePath only when no named spec matches. The
  cache returns the STORED path verbatim (no source-tree
  re-derivation), preserving the `dist/builtins` production layout.
- **Guided Golden Path State Clarity (slice 04.2)** — the
  new-operator golden path said `rig workflow list` shows the
  seeded built-ins, but `rig workflow list` lists workflow
  INSTANCES; the built-in / spec discovery surface is
  `rig workflow specs` (built-ins tagged `(built-in)`). A fresh
  operator following the ordered path before `workflow instantiate
  <name>` was sent to the wrong command. Corrected in
  `docs/reference/getting-started.md` and in the CLI's
  `goldenPathNextSteps()` step 4.
- **For-You Manage-By-Exception (slice 20)** — feed-interaction UX
  on existing signals only (the 0.4.0 attention-detection layer is
  untouched):
  - **Drill-to-terminal** — new `FeedCardTerminalDrill` opens the
    resolved source/author session's terminal PREVIEW directly
    from a card, by reusing `TerminalPreviewPopover` over
    `GET /api/sessions/:sessionName/preview`. Honest framing —
    "terminal preview" / "captured snapshot, not live", never
    claims live state; when no session resolves, the drill renders
    DISABLED with an honest title rather than an empty terminal.
  - **Decision-band sort** — a stable two-band partition lifts
    action-required + approval cards above progress / observation /
    shipped within the existing newest-first comparator. NOT a
    ranking engine (no scores, no card mutation). Applied at the
    single Feed merge-consumption seam.
  - **One-click approve (approve-only)** — `VerbActions` gains an
    additive `oneClickVerbs` prop. APPROVE submits directly on
    click; deny/route stay select+confirm. Defense in depth: the
    prop type narrows to `Extract<MissionControlVerb, "approve">`
    so misuse is a compile error, and a runtime
    `ONE_CLICK_SAFE_VERBS` allowlist rejects cast-forced
    `"deny"`/`"route"` at the call site. Reuses the same
    `performSubmit` mutation path (identical optimistic receipt
    and held-error behavior).
- **CLI Release-Surface Parser (slice 13.1)** — deterministic TypeScript
  Compiler API extractor that walks the Commander registrations in
  `packages/cli/src/commands/*.ts` at two git refs and emits a
  structured release-surface-diff of added commands, added flags, and
  removed/renamed surface. Resolves both Commander idioms (chained
  inline subcommands and factory indirection). Emits the
  REGISTRATION name (`rig-policy.ts` surfaces as `policy`); option
  name-tokens are taken from `.option()` arg[0] only, so
  template-literal descriptions never drop an option. Reads batched
  via one `git cat-file --batch` per ref (fast, offline,
  deterministic). 3-part honest failure shape; never a silent empty
  diff. Not registered as a `rig` verb at 0.3.3 (a separate decision)
  — the module is product code with hermetic test fixtures
  (including a v0.3.1..v0.3.2 worked example).
- **Skill <-> CLI-Surface Binding Index (slice 13.2)** — deterministic
  offline lookup that joins a release surface-diff to "which skills
  are affected by this release". Composes with the slice 13.1 parser;
  invocation-agnostic. Implements the ratified join grammar:
  component-wise prefix match in either direction (so `up` does not
  match `update`); conservative over-include bias for the
  no-false-negative floor. Drops `--version` from the binding index
  (the canonical grammar is command paths only, not global flags);
  the skill body still documents `rig --version` as prose. Ships
  with the v0.3.2-affected-skills regression fixture derived from
  the corpus.

## User-Facing Changes

### `rig archive` and `rig unarchive`

New top-level commands. Archive is non-destructive: the rig row is
preserved, an `archived_at` timestamp is set, and a `rig.archived`
event fires.

```
rig archive <rigId> [--force] [--json]
rig unarchive <rigId> [--json]
```

`rig unarchive` is always safe (no `--force`, no guard). `rig
archive` against a running or degraded rig surfaces a 3-part
fact / consequence / action error and accepts `--force` to
override.

Default reads exclude archived rigs:

```
rig ps [--include-archived]
GET /api/rigs/summary?includeArchived=true
GET /api/rigs/summary?archived=only
GET /api/ps?includeArchived=true
```

`rig ps` marks archived rows with `*` and renders a legend. The
`--include-archived` flag propagates through cross-host argv.

`rig up <name>` against an archived-only name is refused honestly
with an error pointing at `rig unarchive` (no silent restore, no
silent fall-through); `--json` form emits `rig_archived`.

UI surfaces a lazy collapsible "Archive" section under the
localhost host node. The section is fed by a separate archived-only
query (`useArchivedRigs`, `?archived=only`), not by client-side
filtering of the default list. `rig.archived` / `rig.unarchived`
events drive global query invalidation
(`["rigs","summary"]` + `["rigs","summary","archived"]` + `["ps"]`)
so a CLI archive reactively refreshes other mounted UIs.

### `rig add` and `rig add-member`

New top-level command for the `add_member` converge op. Adds a
member to an existing pod in a live rig from a YAML/JSON member
fragment file:

```
rig add <rigId> <podNamespace> <member-fragment-path> \
    [--json] [--rig-root <root>]
```

The member fragment tolerates a bare member or a `{ member }`
wrapper. Field names follow the spec snake_case shape (`id`,
`runtime`, `agent_ref`, `profile`, `cwd`, ...). Optional
pod-local `edges` (array of `{from, to, kind}`) is preserved
through to the converge op:

- Present-but-non-array `edges` is REJECTED (not silently dropped):
  CLI prints a clear error and exits 1 before POST; route returns
  400 `validation_failed`; domain also returns `validation_failed`
  as defense in depth.
- Edge `kind` is validated against the canonical `VALID_EDGE_KINDS`
  set exported from `rigspec-schema.ts` and reused (not duplicated).
  `kind: nonsense` returns 400 `validation_failed` naming the valid
  set; no edge is persisted.

Daemon route: `POST /api/rigs/:rigId/pods/:podNamespace/members`.
Outcome -> HTTP mapping: `rig_not_found` / `pod_not_found` -> 404;
`member_conflict` -> 409; `validation_failed` /
`preflight_failed` -> 400; success -> 201 (with per-node launch
status in the body). MCP tool `rig_add` ships in lockstep.

### `rig down` Accepts Name-or-Id

`rig down <name>` now works. The v0.2.0-era `rig down <name>` HTTP
404 caveat is gone.

```
rig down <rigId-or-name> [--delete] [--json]
```

Resolver semantics:

- `/api/rigs/summary?includeArchived=true` is fetched, then:
- id-match across ALL rigs (archived ids still reach teardown);
- name-match ACTIVE rigs only (`archivedAt == null`).

Same-name active+archived pair resolves the ACTIVE rig (not
ambiguous). Archived-only name does not resolve by name (use the
id, or `rig unarchive` first).

### `rig send --body-file -` for Stdin

Inline payload delivery from a file or stdin, replacing the prior
temp-file folklore:

```
rig send <session> --body-file <path> [--verify] [--json]
rig send <session> --body-file - [--verify] [--json]   # stdin
```

`-` reads from stdin. Mutually exclusive with `--body <text>`.
Kills the backtick-shell-corruption class for multiline send
bodies. Companion to the slice 16 / slice 17 transport fixes so
the surface no longer caps at the OS argv limit and no longer
breaks on dash-prefixed content.

### `rig send --verify` Delivery Outcomes

`--verify` responses now name the outcome explicitly:

- `delivered` — verified succeeded.
- `rendered-unconfirmed` — text + Enter both succeeded but the
  post-send capture could not re-confirm (TUI redraw race, or
  capture threw). The message LANDED; the render is not
  re-confirmable.
- `failed` — the transport itself failed.

The legacy `Verified: yes/no` line is preserved verbatim
(parsers); a new `Delivery:` line carries the named outcome plus,
for `rendered-unconfirmed`, a `rig capture <session>`
confirmation pointer. `--json` carries the `outcome` field
through.

`exit` codes: `delivered` and `rendered-unconfirmed` exit 0; the
middle is not dressed as success or failure. `failed` exits
non-zero with HTTP mapping unchanged. Mid-work / wait-for-idle
REFUSALS deliberately carry no outcome (nothing was sent; a
refusal is not a failed delivery).

### `rig whoami --json` Peers Contract

`peers[]` is this rig's roster EXCLUDING self. It is NOT a
directionally-edged subset, and it is NOT host inventory.

- For the directional graph, read `edges{}`.
- For the inventory including self + live state, use
  `rig ps --nodes`.

`WhoamiResult.peersNote` carries the contract in-band. The CLI
Peers header keeps the literal `Peers:` prefix verbatim (shipped
parsers grep on it) and adds the clarifier pointing at `edges`
and `rig ps --nodes`. `peers[]` name and shape are untouched;
this is an additive clarification only.

### Workflow Instantiate By Name

`rig workflow instantiate <name>` now resolves built-in names
end-to-end:

```
rig workflow instantiate <name-or-source-path> [--cwd <path>] [--json]
```

Built-in names (`conveyor`, `basic-loop`, ...) are resolved
against the seeded spec cache BY NAME first; literal source paths
remain a fallback for unmatched names (honest 404 if neither
matches). Closes the discovery / use divergence where
`rig workflow specs` advertised names that `rig workflow
instantiate` could not consume.

### Guided Golden Path Discovery Surface

The new-operator golden path now points at the correct discovery
verb:

- `rig workflow specs` lists built-in / seeded workflow specs
  (`(built-in)` tagged). USE THIS to discover names before
  `rig workflow instantiate <name>`.
- `rig workflow list` lists workflow INSTANCES (created from
  specs). Empty for a fresh operator.

Corrected in `docs/reference/getting-started.md` and in the
CLI's `setup` golden-path text.

### For-You Manage-By-Exception

Feed-interaction UX on existing signals (the 0.4.0
attention-detection layer is untouched):

- **Drill to terminal preview** — each card's footer carries a
  drill that opens the resolved source/author session's terminal
  PREVIEW via `GET /api/sessions/:sessionName/preview`. Labelled
  "terminal preview" / "captured snapshot, not live" — never
  claims live state. Disabled with an honest title when no
  session resolves.
- **Decision-band sort** — action-required + approval cards lift
  above progress / observation / shipped within the existing
  newest-first comparator. Stable two-band partition; not a
  ranking engine.
- **One-click approve** — APPROVE submits directly on click; deny
  and route stay select+confirm. Compile-time + runtime guards
  prevent the one-click path from being applied to non-approve
  verbs (`ONE_CLICK_SAFE_VERBS = {"approve"}`).

## Operator / Setup Notes

After upgrading the published CLI:

```bash
npm install -g @openrig/cli@0.3.3
rig --version    # expect 0.3.3
rig daemon stop
rig daemon start
```

Migration `042_rig_archive` applies automatically on first
post-upgrade daemon start. It is `ALTER TABLE rigs ADD COLUMN
archived_at TEXT` plus `idx_rigs_archived` — append-only,
non-destructive, no rigs-row restructure / rebuild / drop.
Existing rows are unaffected.

For the new `rig archive` surface: default reads still exclude
archived rigs at the daemon layer. With no rigs archived, no
behavior change. `rig ps --include-archived`, the
`?includeArchived=true` / `?archived=only` query params, and the
UI's lazy Archive section opt archived rigs back in.

For the new `rig add` surface: the member fragment file is a YAML
or JSON document using the rig spec's snake_case field names
(same authoring format as a `RigSpec` pod member). The MCP
`rig_add` tool accepts the same fragment shape.

For `rig down <name>`: previously-documented agent workarounds
that resolved the rig id by hand before calling `rig down` no
longer need to do that step. The id-based form continues to work
unchanged.
