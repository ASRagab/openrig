# OpenRig v0.3.4

## Summary

OpenRig v0.3.4 is a recovery + resilience release. Two new top-level CLI
verbs land (`rig start` and `rig reconcile-session`), `rig up` flips
from fresh-by-default to resume-original-by-default with `--fresh
<seats...>` as the explicit per-seat opt-in, the restore status surface
collapses every honest outcome into a named five-term vocabulary
(`resumed` / `fresh-primed` / `awaiting-decision` / `attention_required`
/ `failed`) across `rig up` / `rig restore` / `rig ps`, and a periodic
snapshot scheduler ships as the crash-insurance floor under
event-driven and teardown snapshots. A handful of long-standing
recovery paper cuts are retired in the same cut: pod-aware narrow
launch leaves dead-end-land for a node-granular managed partial
restore through `rig launch`, a stuck `attention_required` seat now
clears through an evidence-gated `rig seat clear-attention` rather
than hand-edited SQLite, the Codex profile-v2 preflight stops
misclassifying invalid TOML as a legacy-profile reason, the cmux
launch-readiness gate surfaces honest partial workspace state, and a
slow Codex resume no longer gets mis-categorized.

No new schema-bumping migrations in 0.3.4. Existing v0.3.3 databases
upgrade by running `rig daemon start` on the new daemon.

## Included In This Release

- **`rig start` (slice 01)** — new top-level recovery entry point.
  One command that sequences daemon-start, kernel readiness wait,
  restore-usable candidate listing, picker or headless flag
  selection, and per-rig restore + reconcile. Re-codes nothing
  underneath: it sequences existing primitives. Flags `--last`,
  `--all`, `--rigs <names...>`, `--json`. Candidate listing and
  apply are both id-grounded against `/api/rigs/:id/up` (the
  same-name-rigs class never reaches the name-based `/api/up`
  route).
- **`rig up` Resume-By-Default (slice 02)** — `rig up <existing-rig>`
  now resumes original sessions by default. `--fresh <seats...>`
  becomes the explicit per-seat opt-in for operation B (deliberate
  fresh-prime). Previously the default was fresh-prime; agents had
  to know to ask for resume out-of-band. The seat-level outcome
  rides the five-term vocabulary below; `awaiting-decision` is the
  honest zero-session state when the daemon cannot resume and the
  operator has not opted into fresh.
- **`rig reconcile-session` (slice 03)** — new top-level command.
  Adopts a LIVE, hand-resumed canonical session back into its
  persisted node without launching, relaunching, killing, replaying
  startup, pressing resume menus, compacting, or typing into the
  pane. The repair for "I manually resumed the seat; the daemon
  still shows it down." Same node id, no re-key; reports projection
  drift honestly; conversation continuity is never claimed.
- **`rig up --plan` Read-Only Restore Preview (slice 04)** —
  `rig up <existing-rig> --plan` returns a read-only restore
  preview keyed by the five-term vocabulary and an honest async
  timeout. No mutation; no silent claims of resumability when the
  daemon cannot prove it.
- **Five-Term Restore Status Vocabulary (slice 06)** — the prior
  2-3 term collapsed model is replaced across `rig up`,
  `rig restore`, and `rig ps`:
  - `resumed` — original session continuity proven.
  - `fresh-primed` — deliberate fresh-prime (operation B); a brand
    new session was started instead of resumed.
  - `awaiting-decision` — daemon could not resume and operator has
    not opted into fresh; ZERO session started. The honest
    zero-session state, not a silent failure.
  - `attention_required` — seat is live or parked but needs
    operator action (auth gate, model-selection menu, trust
    prompt, stuck recovery). NEVER `failed`.
  - `failed` — the launch transport itself failed.
- **Codex Profile-V2 Preflight (slice 07)** — profile-load check
  on profile-bearing launch and restore surfaces. Legacy-profile
  detection narrowed to legacy-specific patterns only (`contains
  legacy`, `[profiles.<name>]`, `cannot be used while.*legacy`,
  `legacy profile selector`); the prior generic match against
  `failed to load configuration` was also catching invalid-TOML
  stderr and giving the wrong migration hint. Invalid TOML now
  surfaces the actual parse reason (up to 3 stderr lines) with a
  TOML-validity hint.
- **cmux Launch Readiness (slice 08)** — launch path no longer
  silently produces a partial cmux workspace. When cmux comes up
  with a subset of expected windows / panes, the seat surfaces an
  honest partial-workspace state with a one-click open-missing
  affordance instead of being reported as launched-clean.
- **Periodic Snapshots (slice 09)** — `snapshots.periodic` config
  family plus a daemon-side `PeriodicSnapshotScheduler`. Every
  enabled rig captures an `auto-periodic` snapshot on interval and
  the repository prunes by retention-keep. Provides a true
  crash-insurance floor — `auto-periodic` and `auto-pre-down`
  compete on freshness in the same restore selector (the newer
  wins), so a daemon crash between teardown snapshots no longer
  costs an arbitrary amount of lifecycle state. Last-snapshot /
  floor surfaced in `rig ps` / status.
- **`rig seat clear-attention` (slice 10)** — new subcommand on
  `rig seat`. Evidence-gated, operator-attested, audited reconcile
  of a stuck `attention_required` seat back to `ready`. Replaces
  the prior hand-edit-SQLite / fake-clear pattern that agents
  reached for when a seat got stuck. The audit row records who
  cleared, from-state, the evidence path, and any operator
  attestation; `--reason <text>` is the operator-attestation
  override for the evidence gate.
- **Node-Granular Managed Partial Restore (slice 11)** — `rig
  launch <rigId> <nodeRef>` and `rig launch <rigId> --seats <a,b>`
  relaunch a seat or subset by logical id through the
  orchestration layer. Retires the v0.3.x-era
  `pod_aware_launch_unsupported` dead-end and the
  "use rig up instead" workaround that lost per-seat control.
  Held seats carry a `held-reason`; failed targets surface as
  `failed (liveness unknown)`; already-running seats are reported,
  not relaunched.
- **Slow Codex Resume Classification Fix (slice 13)** — internal
  classification tweak for slow-resume Codex sessions: a seat that
  was genuinely resuming but slowly was previously mis-categorized
  by `verifyResumeLaunch`. Now correctly classified through the
  same five-term vocabulary.
- **Pod-Aware Claude Resume-Selection Menu (slice 05 rev1)** —
  follow-on to 0.3.3 slice 21 FR-2: `ClaudeCodeAdapter.verifyResumeLaunch`
  now treats a Claude resume-selection menu the same way it treats
  the Codex unresolved-gate case — returns immediately with
  `recovery: attention_required` and last-12-line pane evidence,
  never sends a numeric selection keystroke, and never lets the
  loop time out to a generic failure.

## User-Facing Changes

### `rig start` — Recovery Entry Point

New top-level command. Starts the daemon, waits for the kernel, lists
restore-usable rigs, and applies restore + reconcile through the
existing primitives:

```
rig start                            Interactive: daemon + kernel + pick-and-restore
rig start --last                     Headless: restore all rigs that were last running
rig start --all                      Headless: restore all rigs with restore-usable snapshots
rig start --rigs <name> [<name>...]  Headless: restore only the named rigs
rig start --json                     JSON output for agents
```

Listing and apply are both id-grounded against
`POST /api/rigs/:id/up`. Same-name rigs surface as separate
candidates; selection carries the candidate's `rigId` end-to-end so
the name-based `/api/up` route is never consulted (no
`ambiguous_name` 409 in the recovery path).

`rig start` is sequencing-only; it does not invent recovery
semantics. The per-rig restore call is the same `/api/rigs/:id/up`
that `rig up <existing>` uses, and seat-level outcomes carry the
same five-term vocabulary.

### `rig up` — Resume Original By Default

`rig up <existing-rig>` resumes original sessions by default.
`--fresh <seats...>` is the per-seat opt-in for deliberate
fresh-prime (operation B):

```
rig up <existing-rig>                       Resume original sessions (default)
rig up --existing <name> --fresh dev-impl   Deliberately fresh-prime dev-impl only
rig up <existing-rig> --plan                Read-only restore preview (no mutation)
```

- Seats whose original session resumes report `resumed`.
- Seats opted into `--fresh` report `fresh-primed`.
- Seats the daemon cannot resume and the operator has not opted
  into fresh report `awaiting-decision` — ZERO session started.
  Headless callers see the explicit hint `rig up --existing
  <source> --fresh <logical-id>`; TTY callers are offered a
  per-seat `[y/N]` prompt before any fresh-prime is started.
- Seats live or parked but needing operator action (auth gate,
  model selection, trust prompt, stuck recovery) report
  `attention_required` — never silently re-classified as `failed`.
- Only transport-level failure reports `failed`.

The prior fresh-by-default behavior is gone. If a workflow relied on
implicit fresh, name `--fresh <seats...>` explicitly.

### `rig up --plan` — Read-Only Restore Preview

```
rig up <existing-rig> --plan [--json]
```

Returns the restore plan without mutating: per-node intended action
(resume / fresh-prime / awaiting-decision), the snapshot the plan
would consume, and whether the current state would be captured before
mutation. The plan path carries an honest async timeout — when the
preview cannot complete within bound, the response says so rather
than claiming a clean plan.

### `rig reconcile-session` — Adopt a Hand-Resumed Session

New top-level command. Use after manually resuming a seat (e.g.
`claude --resume <id>` / `codex resume <id>`) inside its canonical
tmux session: the daemon still shows the seat down. Reconcile binds
the live process to its OWN persisted node and updates the
projection so `rig ps` / topology / `rig send` / `rig capture` /
queue routing work again. It NEVER launches, relaunches, kills,
replays startup, presses resume menus, compacts, or types into the
pane.

```
rig reconcile-session <session>                                   Auto-resolve target node
rig reconcile-session <session> --rig <rigId> --node <logicalId>  Disambiguate
rig reconcile-session <session> --no-launch                       Explicit (default mode)
rig reconcile-session <session> --json                            JSON output for agents
```

`--rig` and `--node` must be provided together (disambiguation).
Same node id, no re-key. Anything the reconcile could not prove is
reported as projection drift (a list of unproven metadata fields);
conversation continuity is reported as a status string, never
claimed as proven.

### Five-Term Restore Status Vocabulary

The seat-level status field on `rig up`, `rig restore`, and `rig ps`
now carries one of five values:

| Status                | Meaning                                                                                            |
| --------------------- | -------------------------------------------------------------------------------------------------- |
| `resumed`             | Original session continuity proven by the native-resume probe.                                     |
| `fresh-primed`        | Operator opted into a deliberate fresh-prime (operation B); a brand new session started.           |
| `awaiting-decision`   | Daemon could not resume and operator has not opted into fresh. ZERO session started.               |
| `attention_required`  | Seat is live or parked but needs operator action (auth, model selection, trust, stuck recovery).   |
| `failed`              | The launch transport itself failed.                                                                |

`awaiting-decision` is new and replaces the prior pattern where a
zero-session outcome was either misreported as success or as
failure. `attention_required` covers the broader live-but-stuck
class; a live or parked seat is NEVER reported as `failed`.

### `rig seat clear-attention` — Audited Stuck-Seat Recovery

Subcommand on `rig seat`. Evidence-gated, operator-attested, audited
reconcile of a stuck `attention_required` seat back to `ready`:

```
rig seat clear-attention <session>
rig seat clear-attention <session> --reason "<operator attestation>"
rig seat clear-attention <session> --json
```

- Without `--reason`, the clear requires evidence the seat is
  actually back to a clean state (the daemon-side gate).
- `--reason <text>` is the operator-attestation override; the
  audit row records the attestation string verbatim.
- Output: `Cleared <session>: <from> -> ready (<clearedBy>)`. The
  audit log captures session, from-state, gate-or-attestation, and
  operator address.

Replaces the prior hand-edit-SQLite / fake-clear pattern. The seat's
node id is unchanged.

### Node-Granular Managed Partial Restore

`rig launch` works for single-seat and subset relaunch on a running
rig through the orchestration layer:

```
rig launch <rigId> <nodeRef>                Single seat (logical id or node id)
rig launch <rigId> --seats <a,b,c>          Subset (comma-separated logical ids)
rig launch <rigId> --seats <a,b> --hold-reason "<text>"  Reason for held non-targets
rig launch <rigId> --seats <a,b> --json
```

- Single-target: `POST /api/rigs/:rigId/nodes/:nodeRef/launch`.
- Subset: `POST /api/rigs/:rigId/nodes/launch-subset`. Reports
  `launched`, `held` (with reason), `alreadyRunning`, and
  `failedTargets` (liveness unknown) separately — partial outcomes
  are never collapsed.

Retires the `pod_aware_launch_unsupported` dead-end and the
"use rig up instead" workaround. Use `rig launch` for per-seat or
per-subset relaunch on a running rig; `rig up <existing>` remains
the whole-rig restore entry.

### Periodic Snapshots

The crash-insurance floor under event-driven and teardown
snapshots. New `snapshots.periodic` settings family:

| Key                                    | Default | Meaning                                    |
| -------------------------------------- | ------- | ------------------------------------------ |
| `snapshots.periodic.enabled`           | `true`  | Master switch                              |
| `snapshots.periodic.interval_seconds`  | `300`   | Snapshot interval per enabled rig          |
| `snapshots.periodic.retention_keep`    | `10`    | Per-rig `auto-periodic` retention count    |

Snapshots are captured with kind `auto-periodic`; the repository
prunes by `retention_keep` per rig. Restore selection treats
`auto-periodic` and `auto-pre-down` symmetrically — the newer of
the two wins, so a daemon crash between teardown snapshots no
longer loses arbitrary lifecycle state. Last-snapshot / floor is
surfaced in `rig ps` / status so an operator can see the floor
at a glance.

Tune via the settings store (`rig config set ...`) or rely on
defaults; the scheduler reads at daemon startup and respects
`snapshots.periodic.enabled = false` as a master switch.

### Codex Profile-V2 Preflight

The profile-load check on profile-bearing launch and restore
surfaces is now honest about WHICH failure it sees:

- **Legacy-profile detection narrowed**: matches `contains legacy`,
  `[profiles.<name>]`, `cannot be used while.*legacy`, and `legacy
  profile selector` only. The prior generic match against `failed
  to load configuration` was also catching invalid-TOML stderr.
- **Invalid TOML surfaces the parse reason**: up to 3 stderr lines
  (e.g. `expected newline`), with a TOML-validity hint instead of
  a wrong migration hint pointing at `[profiles.<profile>]`.

The auth-refusal projection path is unchanged; this is a
classification fix in the preflight discriminator.

### cmux Launch Readiness

The cmux launch path no longer silently produces a partial
workspace. When cmux comes up with a subset of expected windows or
panes, the seat surfaces an honest partial-workspace state with a
one-click open-missing affordance instead of being projected as
launched-clean.

### Slow Codex Resume Classification

Internal classification tweak in `verifyResumeLaunch` for slow but
genuinely-resuming Codex sessions. Previously the slow path got
mis-categorized through the unresolved-gate branch; now it
correctly classifies through the same five-term vocabulary and
reaches `resumed` when the readiness probe confirms it.

### Pod-Aware Claude Resume-Selection Menu

Follow-on to 0.3.3 slice 21 FR-2. `ClaudeCodeAdapter.verifyResumeLaunch`
now treats a Claude resume-selection menu the same way it treats
the Codex unresolved-gate case: returns immediately with
`recovery: attention_required` and last-12-line pane evidence on
both the inner poll loop and the final-probe exit path. Zero
numeric selection keystrokes are ever sent.

## What to STOP Using

These patterns are retired by 0.3.4. Stop reaching for them; the
named replacement is the supported path:

- **STOP** treating `rig up <existing-rig>` as fresh-by-default.
  Resume-original is now default. Use `--fresh <seats...>` when a
  deliberate fresh-prime is the goal (operation B).
- **STOP** reading a stuck or parked seat as `failed`. A live or
  parked seat carries `attention_required`; only the transport
  itself failing is `failed`. The honest zero-session state is
  `awaiting-decision`, not `failed`.
- **STOP** reaching for `pod_aware_launch_unsupported` workarounds
  or "use `rig up` instead" for per-seat relaunch. Use `rig launch
  <rigId> <nodeRef>` or `rig launch <rigId> --seats <a,b>`.
- **STOP** hand-editing SQLite or fake-clearing a stuck
  `attention_required` seat. Use `rig seat clear-attention
  <session>` (evidence-gated) or `rig seat clear-attention
  <session> --reason "<attestation>"` (operator-attested,
  audited).
- **STOP** relying only on event-driven or teardown snapshots for
  crash safety. The periodic snapshot scheduler is the floor;
  tune `snapshots.periodic.interval_seconds` /
  `snapshots.periodic.retention_keep` to taste.
- **STOP** classifying invalid TOML as a legacy-profile problem.
  The Codex profile-v2 preflight now surfaces the actual TOML
  parse reason; legacy detection requires legacy-specific
  patterns.
- **STOP** restoring a hand-resumed session by relaunching the
  whole rig. Use `rig reconcile-session <session> --no-launch` —
  no relaunch, no input written, same node id.
- **STOP** projecting a partial cmux workspace as launched-clean.
  The launch-readiness gate surfaces partial state honestly with
  a one-click open-missing path.

## Operator / Setup Notes

After upgrading the published CLI:

```bash
npm install -g @openrig/cli@0.3.4
rig --version    # expect 0.3.4
rig daemon stop
rig daemon start
```

No new schema-bumping migrations in 0.3.4. The
`snapshots.periodic.*` settings have built-in defaults
(`enabled=true`, `interval_seconds=300`, `retention_keep=10`); no
operator action is required to pick up the crash-insurance floor.

Operators installing v0.3.4 fresh from npm have full access to the
new surfaces immediately. Operators upgrading an existing host
follow the standard daemon-restart flow above; the 0.3.4 host-
upgrade flow is held for separate sequencing and is NOT a
prerequisite for fresh-install access.

For the new `rig start` surface: it sequences existing primitives
(daemon + kernel + per-rig `/api/rigs/:id/up`). The picker is TTY-
interactive by default; the headless flags `--last`, `--all`, and
`--rigs <names...>` cover scripted recovery.

For `rig up` resume-original-by-default: any agent or script that
implicitly depended on fresh-prime as the default must now name
`--fresh <seats...>` explicitly. The five-term vocabulary applies
on the response; treat `awaiting-decision` as the headless
"operator must decide" outcome, not an error.

For `rig seat clear-attention`: prefer evidence-gated clear when
the seat is genuinely back. The `--reason <text>` override is the
audited operator-attestation path; the attestation string lands in
the audit row verbatim.

For periodic snapshots: tune `snapshots.periodic.interval_seconds`
down for higher-churn rigs and `snapshots.periodic.retention_keep`
up for longer recovery windows. The default 5-minute interval +
10-snapshot retention is a balanced floor.
